# Run Behaviour Specification

## Purpose

`Run` submits one generated batch (`seedBase`) to Slurm using metadata stored in SQLite.
Submission is split per node: one manifest and one Slurm script per selected node.

## Inputs

- `seedBase` selected from generated batches.
- Generated run metadata from `generated_runs`.
- Slurm/TOPAS settings from configuration:
  - `Tsebt.Topas.Executable`
- `Tsebt.Slurm.Partition`
- `Tsebt.Slurm.CpusPerTask`
- `Tsebt.Slurm.Account` (optional)

## Paths

Run folder:

```text
{AppRoot}/{Paths.Runs}/{seedBase}
```

Generated by Run (examples):

```text
{AppRoot}/{Paths.Runs}/{seedBase}/run_manifest_monte-carlo-01.tsv
{AppRoot}/{Paths.Runs}/{seedBase}/run_batch_monte-carlo-01.slurm
{AppRoot}/{Paths.Runs}/{seedBase}/run_manifest_monte-carlo-02.tsv
{AppRoot}/{Paths.Runs}/{seedBase}/run_batch_monte-carlo-02.slurm
```

## Manifest format

Header:

```text
task_id\tnode_name\trun_id\tinput_file_path\tlog_file_path
```

Each node manifest contains only rows for that node. Task ids restart at 1 per node.
Each row contains:

- task id (1-based)
- node name (resolved from configured nodes by node digit)
- run id
- input file path
- log file path (`runs/{seedBase}/{runId}.log`)

## Preflight checks

Preflight verifies:

- generated runs exist
- batch status is `Generated`
- no previous Slurm job id
- all input files exist
- run folder exists
- no output base collisions
- no output CSV collisions
- no output log collisions

If any check fails, submission is blocked.

## Script behavior

Each node Slurm script:

- defines `TOPAS` executable variable
- defines `MANIFEST` path variable
- resolves row by `SLURM_ARRAY_TASK_ID`
- reads `TASK_ID NODE_NAME RUN_ID INPUT_FILE LOG_FILE`
- sets:
  - `#SBATCH --nodelist={nodeName}`
  - `#SBATCH --chdir={AppRoot}/{Paths.Runs}/{seedBase}`
  - `#SBATCH --account={account}` when account is non-empty
- runs:

```bash
"$TOPAS" "$INPUT_FILE" > "$LOG_FILE" 2>&1
```

## Submission flow

`submitRun` performs:

1. Load batch details.
2. Block already-submitted batches (`Submitted/Running/Completed/Failed` with job id).
3. Run preflight.
4. Build per-node manifests and scripts.
5. Write all manifest/script files.
6. Execute `sbatch` once per node script.
7. Parse each Slurm job id from output (`Submitted batch job 12345`).
8. Update SQLite batch metadata (`run_status`, `slurm_job_id`, paths, timestamp), where `slurm_job_id` stores comma-separated ids.
9. Update generated runs status to `Submitted`.

If any `sbatch` fails, files remain for debugging and batch status is not marked `Submitted`.

## UI behavior notes

- Server stores full absolute paths.
- Run wizard UI displays AppRoot-relative paths for readability where possible.
